# ADR-008: Internal Package Architecture

**Status:** Accepted
**Date:** 2026-04-12
**Principles:** P5, P2

## Context

bukeer-studio contains two internal packages that form the domain layer:

- `@bukeer/theme-sdk` — Design token system, M3 theme compilation, tourism presets
- `@bukeer/website-contract` — Zod schemas, TypeScript types, section type constants

These packages are consumed by the Next.js app and potentially by external tools (CLI, validators). The architecture must support both monorepo development (fast iteration) and standalone consumption (npm publish).

## Decision

### Package dependency graph (strict)

```
@bukeer/theme-sdk (v3.0.0)
    ↓ imported by
@bukeer/website-contract (v0.1.0)
    ↓ imported by
Next.js app (web root)
```

**Hard rules:**
- The app never imports from package internals — only from the barrel export (`index.ts`)
- `website-contract` may import from `theme-sdk`
- `theme-sdk` never imports from `website-contract`
- Neither package imports from the app
- Circular dependencies are a CI-fail violation

### Dual consumption mode

**Monorepo (development):**
```typescript
// next.config.ts
transpilePackages: ['@bukeer/theme-sdk', '@bukeer/website-contract']

// tsconfig.json paths
"@bukeer/website-contract": ["./packages/website-contract/src"],
"@bukeer/theme-sdk": ["./packages/theme-sdk/src"]
```

Next.js transpiles directly from `src/`. No build step needed during development. Changes are picked up immediately by the dev server.

**Standalone (external tools):**
```bash
cd packages/theme-sdk && npm run build    # → dist/
cd packages/website-contract && npm run build  # → dist/
```

The `npm run build` compiles to `dist/` for standalone TypeScript consumers. The `main` and `types` fields in `package.json` point to `dist/`.

### Barrel exports

Each package has a single `src/index.ts` that explicitly exports its public API:

```typescript
// packages/theme-sdk/src/index.ts
export { compileTheme } from './compiler'
export { TOURISM_PRESETS } from './presets'
export type { ThemeInput, DesignTokens, ThemeProfile, CompiledTheme } from './types'

// packages/website-contract/src/index.ts
export { SECTION_TYPES, COPILOT_ACTION_TYPES } from './constants'
export { QuoteRequestSchema, SectionConfigSchema } from './schemas'
export type { WebsiteData, WebsitePage, WebsiteSection, BlogPost } from './types'
```

Internal modules (helpers, utilities, implementation details) are NOT exported.

### Package responsibilities

**@bukeer/theme-sdk:**
- Design token definitions (colors, typography, spacing, motion, elevation)
- M3 theme compilation (tokens → CSS variables)
- Tourism presets (adventure, luxury, tropical, corporate, boutique, cultural, eco, romantic)
- Target-specific compilation (web, mobile, desktop)
- Zero React dependency — pure TypeScript + Zod

**@bukeer/website-contract:**
- Website data types (WebsiteData, WebsitePage, BlogPost, etc.)
- Section type registry (SECTION_TYPES constant)
- Zod schemas for API validation (quote, section, template)
- Shared constants (copilot actions, default values)
- Imports ThemeInput/DesignTokens from theme-sdk for theme validation

### Versioning

Packages use semver. Breaking changes to the public API require a major version bump. The app's `package.json` pins exact versions for reproducible builds.

```json
// package.json (app root)
"dependencies": {
  "@bukeer/theme-sdk": "file:./packages/theme-sdk",
  "@bukeer/website-contract": "file:./packages/website-contract"
}
```

### Tailwind v4 integration

Design tokens from `theme-sdk` are compiled to CSS custom properties, not Tailwind config. This avoids configuration duplication:

```css
/* Generated by compileTheme() */
:root {
  --color-primary: #1a73e8;
  --color-on-primary: #ffffff;
  --font-display: 'Playfair Display', serif;
  --spacing-section: 5rem;
  /* ... */
}
```

Tailwind v4 consumes these via `var()` references in utility classes and component styles.

## Consequences

- **Fast development** — no build step, instant HMR
- **Clean API boundaries** — barrel exports prevent coupling to internals
- **Standalone-ready** — packages can be published to npm if needed
- **Type safety** — shared types ensure consistency between app and packages
- **Trade-off:** `transpilePackages` recompiles on every dev server restart (adds ~2s)
- **Trade-off:** Two `package.json` files to maintain (package + app root)
