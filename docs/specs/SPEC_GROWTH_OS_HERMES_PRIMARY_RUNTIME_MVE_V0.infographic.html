Growth OS Hermes Primary Runtime MVE v0

Hermes deja de ser sidecar y se convierte en el runtime que mueve el crecimiento.

El objetivo del steel thread es probar un camino completo de produccion: Hermes decide, crea tareas, ejecuta con /goal y delegate_task, publica mediante MCP tools scoped, aprende del resultado y deja a Supabase como producto, auditoria y rollback.

100% Autonomo dentro de las acciones v0 permitidas.
1:1 Un board Hermes por account/website para aislar el runtime.
8 Provider profiles para priorizar con datos frescos y citables.
Hermes Kanban cola primaria nativa
Hermes Profiles identidad + skill bindings
MCP Safety Layer scope, Zod, snapshots
Supabase producto, auditoria y rollback

Que cambia

La migracion elimina la cola rigida como centro del sistema y mueve la ejecucion diaria a Hermes.

Antes

Growth OS executor-led

  • Muchas fases previas: candidate, work item, publication job.
  • Estados pre-ready bloquean el flujo aunque el cambio sea puntual.
  • Hermes produce artifacts, pero no dirige la ejecucion.
  • Supabase participa como runtime queue y como producto.
->
Despues

Hermes primary runtime

  • Hermes Kanban nativo es la cola unica para nuevo trabajo.
  • Perfiles Hermes cargan skills y ejecutan con /goal + delegate_task.
  • El Chief prioriza por impacto esperado al North Star.
  • MCP tools hacen validacion, snapshots, smoke y rollback.
  • Supabase queda como datos reales, auditoria, outcomes y recuperacion.

Steel Thread

El camino minimo que debe pasar en produccion para aceptar el v0.

01 Hermes lee contexto Tenant context, provider profiles, memoria, skills y estado del sitio dentro del scope ColombiaTours.
02 Chief crea tarea La tarea entra al Kanban nativo con baseline, impacto esperado y success metric.
03 /goal delega Chief usa delegate_task para lanzar el perfil con su skill y contexto minimo.
04 Tool aplica cambio MCP Safety valida permisos, crea snapshot, escribe Supabase y corre smoke.
05 Sistema aprende Hermes cierra tarea, registra outcome y usa el aprendizaje en el siguiente ciclo.

Loop de maximizacion Growth OS

Hermes no ejecuta por orden de llegada: prioriza por impacto esperado al North Star.

A Objetivo North Star: qualified trip requests por mes.
B Facts Provider profiles con freshness, confidence y source refs.
C Scoring Impacto esperado, confianza, riesgo, esfuerzo y ventana de medicion.
D Produccion Agente ejecuta por MCP tool con snapshot, smoke y rollback.
E Learning Outcome actualiza memoria/skills y cambia la siguiente decision.

Kanban operativo

Usamos los estados nativos de Hermes; smoke y rollback viven como metadata/eventos.

triage

Existe la tarea; falta clasificar, target o scoring.

todo

Clasificada y asignada; espera dependencias.

ready

Dependencias listas; /goal puede claimar.

running

Profile session Hermes ejecutando.

blocked

MCP Safety, policy, provider freshness o perfil bloquearon.

done

Completada: cambio aplicado, no-op o rollback concluido.

archived

Fuera del tablero activo, conserva historia.

Metadata Growth encima de Hermes

No se crean estados extra; se registran como tool results, comments, snapshots y outcomes.

Tool

verifying

Estado interno de la invocacion mientras corre smoke/auditoria.

Smoke

smoke_passed

Resultado guardado en tool invocation y mutation snapshot.

Recovery

rollback_required

Metadata/comment que normalmente bloquea o dispara rollback.

Audit

rolled_back

Evento de auditoria; la tarea puede terminar en done o archived.

Acciones autonomas v0

Hermes puede actuar sin aprobacion humana solo dentro de estas herramientas.

Produccion hermes_publish_content Publica u optimiza contenido en campos permitidos con profundidad, SEO checks, snapshot y rollback.
Produccion hermes_merge_transcreation Aplica overlays o metadata localizada validando locale pair, calidad y estado hreflang-safe.
Produccion hermes_apply_safe_seo_patch Modifica solo metadata SEO segura en paginas permitidas y verifica render/ruta.
Contexto hermes_refresh_provider_context Refresca contexto de proveedores con cuotas, procedencia y freshness controlado.
Aprendizaje hermes_record_outcome Crea semillas de outcome con metricas, baseline, ventana de evaluacion y refs.
Recuperacion hermes_rollback_change Revierte una mutacion v0 usando rollback payload del mismo account y website.

Que logramos al implementarlo

El valor del MVE no es solo publicar: es operar, medir y mejorar con menos friccion.

Flujo

Menos estados bloqueantes

Nuevo trabajo avanza por Hermes Kanban nativo sin depender de candidates ni publication jobs.

Autonomia

Agentes que actuan

Hermes puede publicar, mergear y aplicar SEO seguro dentro del scope permitido.

Control

Seguridad en tools

La validacion vive en MCP Safety: cuenta, sitio, fields, snapshots, smoke y rollback.

Aprendizaje

Mejora continua

Hermes usa outcomes, memoria y skills para modificar el siguiente trabajo.

Priorizacion

North Star primero

El Chief secuencia tareas por impacto esperado en qualified trip requests.

Guardrails que permanecen

Autonomo no significa sin limites: significa sin aprobacion humana dentro del scope v0.

Bloqueado

Superficies sensibles

Paid media, pricing, disponibilidad, reservas, pagos, CRM masivo, outreach y activacion de experimentos quedan fuera del v0.

Obligatorio

Snapshot y rollback

Cada mutacion productiva debe guardar before snapshot, after payload, smoke result y rollback payload.

Scope

Cuenta y sitio

Toda tool debe fallar cerrada si el account, website, perfil, target o field no coinciden.

Datos

Freshness y source refs

Los provider profiles stale o fuera del allowlist bloquean herramientas que dependan de esos facts.

100% produccion

El MVE solo queda cerrado si todos los gates pasan durante 24 horas continuas.

Runtime

Hermes primario

24h con runtime, MCP tool host y watchdog activos; 0 stuck tasks y 0 duplicados por idempotency key.

Seguridad

Scope cerrado

100% cross-tenant, tools fuera de allowlist y fields prohibidos rechazados; service-role solo en tool host.

Supabase

Auditoria completa

Migracion aplicada desde Flutter, RLS activo en hermes_*, reads por user_roles y 0 writes sin auditoria.

Tools

Acciones reales

1 publish, 1 transcreation merge, 1 safe SEO patch y 1 rollback o dry-verify con snapshots completos.

Contexto

Calidad de entrada

100% tareas productivas con tenant context, provider refs o irrelevancia documentada y freshness PASS.

Score

NO-GO automatico

100% si todos los hard gates pasan; cualquier P0/P1 bloquea produccion primaria.

Regla final del MVE: Hermes decide y ejecuta; MCP Safety valida y aplica; Supabase persiste producto, auditoria y outcomes; Bukeer Studio observa y recupera.
